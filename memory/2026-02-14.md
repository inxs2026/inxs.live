## Gateway Crash Root Cause Identified - ~4:45 PM ✅

**Problem:** Gateway crashed/became unresponsive twice today (12:03 PM and again later). Carlo had to reset connection.

**Investigation:** Checked logs via `journalctl --user -u openclaw-gateway`

**Root Cause Found:** OpenAI embeddings API quota EXHAUSTED
```
"message": "You exceeded your current quota, please check your plan and billing details."
"code": "insufficient_quota"
```

**Timeline of failures:**
- 3:23 PM - embeddings quota exceeded
- 4:14 PM - embeddings quota exceeded  
- 4:23 PM - embeddings quota exceeded

**Impact:** Memory system kept trying to embed files, hitting quota wall, retrying with exponential backoff → eventually caused gateway to hang/become unresponsive.

**Fix Applied:** Switched memory embeddings from OpenAI to LOCAL provider (no API needed)
- Used `gateway config.patch` to update config
- Set `agents.defaults.memorySearch.provider = "local"`
- Set `agents.defaults.memorySearch.fallback = "none"`
- Gateway restarted automatically to apply changes

**Result:** Embeddings now run locally via node-llama-cpp - no more OpenAI quota issues, no more crashes.

**Lesson:** OpenAI quota exhaustion can cause cascading failures in the memory system that eventually hang the entire gateway. Local embeddings = more reliable.

**Config change:**
```json
{
  "agents": {
    "defaults": {
      "memorySearch": {
        "enabled": true,
        "provider": "local",
        "fallback": "none"
      }
    }
  }
}
```

**Status:** ✅ RESOLVED - Gateway stable, local embeddings working

**Carlo emphasized:** REMEMBER how to read DRF forms correctly.

- **ALWAYS read DRF .txt files directly with Read tool**
- **NEVER use grep/awk for DRF parsing** - causes confusion
- Morning line odds (7-1) ≠ post position numbers

**The pattern that causes problems:**
```bash

```

**The pattern that works:**
```bash
Read drf.txt offset=LINE limit=100  # ✅ Clear, accurate
```
